4. 경매 시스템

4.1 경매 유형 3가지

유형 A — 신규 매물 경매 구현 완료

1. 유저가 스트리머 검색 → 매물 없음 (또는 보유자 없는 매물)
2. 유저가 경매 등록 (시작가 5만G 이상) → 최소 시세 50,000G 즉시 차감
3. 대기열에 추가 (선착순)
4. 순서가 오면 경매 시작 (5분)
5-A. 입찰 있음 → 최고 입찰자 낙찰
5-B. 유찰 → 등록자가 시작가에 자동 낙찰 (반드시 보유자 생김)

유형 B — 보유자 승인 경매 구현 완료

1. 유저B가 유저A의 매물에 경매 요청 (requestAuction)
   → 면역 기간 중 요청 불가 / 이미 대기 중인 요청 있으면 불가
2. 유저A에게 내 보유 매물 패널에 승인/거부 버튼 노출 (24시간 기한)
3-A. 승인 (respondToAuctionRequest approve) → 경매 대기열 추가, 시작가는 유저A가 지정
3-B. 거부 (respondToAuctionRequest reject) → 해당 매물 24시간 면역 설정
3-C. 24시간 미회신 → 강제청산 (보유자 시세 100% 환급 + 무보유 전환)

낙찰 처리:
4-A. 낙찰 (입찰자 존재) → 낙찰자에게 매물 이전, 유저A에게 낙찰가 100% 즉시 지급
4-B. 유찰 + 유저B 잔액 충분 → 유저B가 시작가에 자동 낙찰, 유저A에게 시작가 100% 지급
4-C. 유찰 + 유저B 잔액 부족 → 유저A 소유 유지 + 해당 매물 24시간 면역 설정

유형 C — 손절 경매 구현 완료

1. 보유자가 본인 매물 경매 등록 (시작가 5만G 이상) — registerAuction type:"selloff"
   → 검색 결과 또는 내 보유 매물 패널의 "손절 등록" 버튼
   → 매물 isLocked: true 설정
2. 5분 경매 진행 (모든 유저 입찰 가능)
3-A. 낙찰 → 낙찰자에게 매물 이전, 보유자에게 낙찰가 100% 즉시 지급
3-B. 유찰 → 경매 취소 (보유 유지, 시세 변동 없음, 면역 기간 없음)

1. 상점 페이지(shop.html) 접속

4. 구매 확인 모달 표시
   - 아이템명 · 효과 요약 · 차감 G · 구매 후 잔액 미리보기
   - needsTarget 아이템(연장권·면역권)은 대상 매물 선택 드롭다운 표시
     * 강제청산 기간 연장권: 모든 내 보유 매물 표시 (요청 없는 매물 선택 시 예약, 있는 매물 선택 시 즉시 적용)
     * 면역 연장권: 면역 기간 중인 매물만 표시
   - 잔액 부족 시 버튼 비활성화 + 부족 금액 안내
5. 확인 → purchaseShopItem CF 호출 (전 아이템 Firestore 트랜잭션)
   - 가격은 서버 SHOP_ITEMS에서 결정 — 클라이언트 가격 조작 불가
   - 잔액 재확인을 트랜잭션 내부에서 수행 (TOCTOU 방어)
   - 잔액 차감 / 패스형은 users.{passField} 만료 timestamp 갱신 / 연장권은 listing 또는 request 직접 갱신
6. 성공 토스트 + 잔액 즉시 갱신
7. 효과 적용 시점: 즉시 적용형(패스, 면역 연장) vs 자동 적용형(강제청산 연장권 예약)

19.8 화폐 수지 분석

구분방향계산
광고 등록비 (광고주 지출)회수 (deflation)광고주 수 × 기간 × 200,000G
광고 보상 지급 (시청자 수령)발행 (inflation)DAU × 평균 수령 광고 수 × 5,000G
수지 균형 시나리오 예시
광고주 1명이 1일 광고 등록 (200,000G 회수)
DAU 30명이 각 1개 광고 클릭 시 (150,000G 발행)
→ 순 회수 +50,000G / 일

광고주 2명이 1일 등록 (400,000G 회수)
DAU 30명 × 평균 2개 클릭 (300,000G 발행)
→ 순 회수 +100,000G / 일

광고주 수가 시청자 수보다 적으면 순발행 발생. 광고주 유인책(노출 효과)이 수요를 결정함.
