## 3. 게임 컨셉

### 핵심 플레이 루프

1. 유저가 soop 스트리머의 닉네임/ID를 검색
2. 매물 상태에 따라 구매 또는 거래 제의
3. 시세 변동을 통해 수익 실현 또는 손절
4. 보유 종목으로 자산을 키워가며 랭킹 경쟁

### 게임 화폐

- **단위**: G (Gold)
- **최초 매물 가격**: 50,000G
- **표기 원칙**: 절대 "원/달러" 등 실제 통화 단위 사용 금지 (사행성 회피)

## 4. 거래 메커니즘

### 거래 경로 3가지

#### 경로 A: 미보유 매물 구매 (시스템 거래)

- 주인이 없는 스트리머 매물은 누구나 50,000G에 즉시 구매 가능
- 이전에 손절 처리되어 시장에 풀린 매물도 50,000G로 리셋되어 동일 처리

#### 경로 B: 유저 간 거래 (제안 → 승인)

1. 매수 희망자가 보유자에게 가격 제안 (현재 시세 이상)
2. 보유자가 제안 검토
3. 보유자 승인 시 거래 성사
4. 거래 가격이 새로운 시세로 등록

#### 경로 C: 즉시 손절 (시스템 거래)

- 보유자가 매물을 시스템에 즉시 매각
- 시스템이 **현재 시세의 80%** 가격으로 매입
- 매물은 시장에서 소멸되고, 다시 50,000G의 미보유 매물로 시장에 풀림
- **시세는 변동 없이 유지** (시세 안정성 보호)

### 거래 비교 표

| 경로 | 가격 | 처리 방식 | 시세 영향 |
|---|---|---|---|
| 미보유 구매 | 50,000G 고정 | 시스템 즉시 거래 | 변동 없음 |
| 유저 간 거래 | 제안가 (시세 이상) | 보유자 승인 필요 | 거래가가 새 시세 |
| 즉시 손절 | 시세의 80% | 시스템 즉시 매입 | 변동 없음 |

## 5. 시세 시스템

### 시세 결정 룰

- **최초 매물 시세**: 50,000G 고정
- **이후 시세**: 마지막 거래가
- **즉시 손절 시**: 시세 변동 없음 (보유자만 80% 회수)

### 시세 하락 메커니즘

거래 없는 매물의 시세가 자연 감가상각되도록 하여 시장 정체 방지.

| 무거래 기간 | 시세 변화 |
|---|---|
| 7일 경과 | -5% |
| 14일 경과 | -5% (누적 -10%) |
| 21일 경과 | -5% (누적 -15%) |
| 이후 7일마다 | 추가 -5% |
| **하한선** | **50,000G** (최소가격 밑으로 안 내려감) |

**카운트 리셋 트리거:**
- 거래 성사
- 유효 제안 수신 (현재 시세 이상)

### 휴면 매물 강제 회수

| 항목 | 내용 |
|---|---|
| 트리거 | 보유자 30일 미접속 + 14일 무거래 |
| 회수 가격 | 시세의 70% (보유자에게 입금) |
| 회수 후 | 50,000G로 리셋되어 시장에 풀림 |
| 사전 알림 | 미접속 23일째 경고 (앱 내 알림 + 이메일) |

## 6. 유저 등급 시스템

### 차등 정책

| 기능 | 익명 로그인 | 구글 로그인 |
|---|---|---|
| **초기 자산** | 200,000G | 1,000,000G |
| **보유 종목 한도** | 1개 | 5개 |
| **시스템 거래 (구매/손절)** | ✅ | ✅ |
| **거래 제의 송신** | ❌ | ✅ |
| **거래 제의 수신** | 알림만 가능 | ✅ |
| **제의 승인/거절** | ❌ (구글 전환 필요) | ✅ |
| **데이터 영속성** | 브라우저 종속 | 영구 보관 |
| **모드 성격** | 싱글 플레이 (체험판) | 멀티 플레이 |

### 익명 → 구글 전환 시

- 자산 그대로 승계
- 보유 종목 그대로 승계
- 보유 종목 한도 5개로 확장
- **전환 보너스 지급** (금액 미정)

### 보유 한도 룰

- **한도 미달**: 구매/거래 모두 가능
- **한도 가득**: 판매(거래/손절)만 가능, 구매 불가

### 거래 제의 수신 시 (익명 유저)

- 익명 상태에선 알림만 표시
- 승인하려면 구글 로그인 필수
- 제의 유효기간(72시간) 내 전환 안 하면 자동 만료
- 첫 알림 시 강한 모달, 이후엔 배지로 표시 (UX 균형)

### 익명 유저의 한계 (의도된 설계)

- 보유 가능 매물은 사실상 50,000G짜리 미보유/리셋 매물뿐
- 시세 5만G 매물에 다른 유저가 제의를 보내야 거래 가능 → 자연스러운 컨버전 유도
- "체험판" 명시 필수 (UI 안내)

## 7. 어뷰징 방어 시스템

### 우선과제: 자전거래 자산 증식 방어

게임 경제 자체를 파괴하는 가장 위험한 어뷰징 패턴.

### 1단계: 기본 방어선 (출시 시 필수 구현)

#### 거래 수수료 (가장 강력한 방어책)

- **모든 유저 간 거래에 10% 수수료 부과** (판매자 부담)
- 구매자 100,000G 지불 → 판매자 90,000G 수령 → 10,000G 시스템 회수
- 자전거래 반복 시 자산이 줄어듦 → 어뷰징 동기 제거
- 화폐 싱크(인플레이션 방지) 기능 동시 수행

#### 동일 디바이스/IP 거래 차단

- Cloud Functions에서 거래 시점에 디바이스 ID + IP 비교
- 디바이스 ID는 localStorage에 UUID 발급 + Firebase 기록
- 동일 식별자 간 거래는 차단

#### 신규 계정 거래 제한

- 가입 후 24시간: 즉시 손절 불가
- 가입 후 7일: 일일 거래 횟수 제한
- 가입 후 7일: 강제청산 면제 (보호)

#### 손절 쿨다운

- 매물 구매 후 **24시간** 동안 손절 불가
- 펌핑 후 즉시 손절 어뷰징 차단

### 2단계: 행동 패턴 탐지 (운영하면서 추가)

| 항목 | 내용 |
|---|---|
| 순환 거래 탐지 | A→B→C→A 패턴 자동 플래그 |
| 비정상 거래가 탐지 | 시세의 5배 이상 거래 검토 |
| 자산 급증 탐지 | 단기간 X배 증가 자동 플래그 |

### 3단계: 시세 펌핑 대응 (낮은 우선순위)

- 모든 거래 내역 매물 페이지에 공개 (유저 자율 판단)
- 동일 IP 차단으로 직접 펌핑은 1차 방어
- 운영 데이터 보면서 필요시 인상률 제한 추가 검토

## 8. 강제청산 시스템

거래 제의에 무응답 시 매물이 시장에서 잠기는 데드락 문제 해결.

### 작동 방식

```
0시간: 유효 제의 수신 → 즉시 알림
24시간: "곧 강제청산" 1차 경고
48시간: "내일 강제청산" 2차 경고
72시간: 강제청산 실행
```

### 청산 조건

- 제의 가격이 **현재 시세 이상**일 것 (장난 제의 차단)
- 제의자의 자산이 충분할 것
- 보유자가 가입 후 7일 이상일 것

### 청산 가격

```
청산 가격 = max(제의가, 현재 시세)
```

낮은 가격 제의 후 무응답 노리는 어뷰징 차단.

### 동시 제의 처리

- 카운트는 **첫 제의 기준**
- 청산 시점에 **여러 제의 중 최고가** 채택
- 보유자에겐 최선의 결과, 매수자 간 자연스러운 경쟁

### 보호 조항

| 항목 | 내용 |
|---|---|
| 신규 매물 보호 | 매물 구매 후 7일은 강제청산 면제 |
| 시세 급등 보호 | 24시간 내 X배 이상 급등한 매물은 보류 |
| 휴가 모드 | 자발적 부재 표시 시 면제 (월 1회, 최대 7일) |
| 신규 유저 보호 | 가입 후 7일 이내 면제 |

### 응답 처리

| 보유자 행동 | 카운트 영향 |
|---|---|
| 거래 승인 | 거래 성사, 카운트 종료 |
| 명시적 거절 | 해당 제의 카운트 종료 (다른 제의 있으면 그것 기준 재시작) |
| 단순 로그인 | 영향 없음 (의도적 무시 방지) |

## 9. 화폐 발행원 정책

게임 내 G(Gold) 화폐가 발행되는 모든 경로. 인플레이션 방지를 위해 발행량과 회수량(거래세, 손절 페널티)의 균형 유지가 핵심.

### 가입 보너스

| 구분 | 보너스 |
|---|---|
| 익명 가입 | 200,000G |
| 구글 가입 | 1,000,000G |
| 익명 → 구글 전환 | 1,000,000G까지 차액 채우기 |

**전환 보너스 작동 예시:**
- 익명 자산 200,000G에서 전환 → 800,000G 추가 지급 (총 1,000,000G)
- 익명 자산 600,000G에서 전환 → 400,000G 추가 지급 (총 1,000,000G)
- 익명 자산 1,500,000G에서 전환 → 추가 지급 없음 (이미 초과)

### 일일 출석 보상

**구글 유저 전용** (익명 유저는 출석 보상 없음)

| 연속 출석 | 보상 |
|---|---|
| 1~2일차 | 5,000G |
| 3일차 ~ | 10,000G |
| 7일차 (특별) | 30,000G |
| 30일차 (특별) | 100,000G |

**룰:**
- 자정(KST 00:00) 기준 일일 갱신
- 연속 출석 끊기면 1일차로 리셋
- 익명 유저는 출석 보상 없음 → 부계정 양산 방지 + 컨버전 유도

### 광고 시청 보상

**구글 유저 전용** (익명 유저는 광고 보상 없음)

| 항목 | 값 |
|---|---|
| 1회 보상 | 10,000G |
| 일일 한도 | 5회 (총 50,000G) |
| 광고 사이 쿨다운 | 5분 |
| 시청 완료 조건 | 끝까지 시청해야 보상 (스킵 시 0G) |

**룰:**
- 광고는 보상형 영상(rewarded video) 사용
- 24시간 내 신규 가입자는 광고 보상 차단 (어뷰징 방어)
- 익명 유저는 광고 보상 없음

### 튜토리얼 보상 (첫 경험 보상)

**구글 유저 전용**, 1회성 보상

| 미션 | 보상 |
|---|---|
| 첫 매물 구매 (50,000G 미보유 매물) | 10,000G |
| 첫 유저 간 거래 성사 | 30,000G |
| 첫 즉시 손절 경험 | 5,000G |
| 첫 강제청산 경험 (보유자/매수자 무관) | 10,000G |

**목적:** 단순 화폐 발행이 아닌 게임 메커니즘 학습 유도. 유저가 모든 거래 경로를 직접 체험하게 함.

### 보류 항목 (출시 시점 도입 안 함)

| 항목 | 사유 |
|---|---|
| 친구 초대 보상 | 어뷰징 위험 큼. 안정화 후 그로스 단계에서 검토 |
| 매물 보유 패시브 수익 (배당) | soop API 비용 + 인플레 위험. 시즌 2 검토 |

### 일일 발행량 시뮬레이션

#### 활성 구글 유저 1명 기준 (최대치)

| 항목 | 금액 |
|---|---|
| 출석 보상 | 5,000G ~ 30,000G |
| 광고 시청 (5회) | 50,000G |
| **하루 최대 발행** | **약 80,000G** |

#### 활성 익명 유저 1명 기준

| 항목 | 금액 |
|---|---|
| 모든 일일 보상 | **0G** |
| **하루 최대 발행** | **0G** |

### 회수 메커니즘 (참고)

| 항목 | 회수율 |
|---|---|
| 거래 수수료 | 거래액의 10% |
| 즉시 손절 페널티 | 시세의 20% |
| 휴면 매물 강제 회수 | 시세의 30% |

### 운영 모니터링 필수

- 일일 발행량 / 회수량을 실시간 추적하는 운영 대시보드 필수
- 출시 후 1~3개월 데이터 기반으로 보상 액수/거래세 조정
- 인플레이션 비율이 일정 수준 초과 시 자동 알림

## 11. MVP 범위 및 개발 로드맵

### MVP 정의

**Minimum Viable Product** — 게임의 핵심 루프(검색 → 구매 → 시세 변동 → 거래/손절)가 작동하는지 검증할 최소 기능 제품. 부가 기능은 출시 후 데이터 기반으로 추가.

### MVP 원칙

1. 핵심 게임 루프가 작동해야 함
2. 부가 기능은 나중에 (스킨, 칭호, 친구 시스템 X)
3. 어뷰징 방어와 보안은 처음부터 필수
4. 빠른 출시 후 데이터 기반 개선

### 11.1 Phase 분류

#### Phase 1: MVP 출시 (필수 기능)

**인증/유저**
- Firebase 인증 (익명 + 구글)
- 익명 → 구글 전환 로직
- 가입 보너스 지급
- 마이페이지 (자산/보유 매물)

**매물/거래 핵심**
- 스트리머 매물 검색/등록
- 미보유 매물 구매 (50,000G)
- 거래 제의 송신/수신/승인/거절
- 거래 제의 만료 (72시간)
- 즉시 손절 (시세 80% 회수)
- 시세 갱신 (마지막 거래가 기준)
- 거래 내역 기록
- 보유 한도 체크 (익명 1, 구글 5)

**어뷰징 방어 (출시 시점 필수)**
- 거래 수수료 10%
- 동일 디바이스/IP 거래 차단
- 신규 24시간 손절 차단
- 매물 구매 후 24시간 손절 쿨다운
- Firestore 보안 규칙

**알림**
- 거래 제의 수신 알림 (인앱)

#### Phase 2: 출시 직후 1개월 내

- 시세 감가상각 시스템 (7일/5%)
- 강제청산 시스템 (자동화)
- 일일 출석 보상
- 광고 시청 보상 (광고 SDK 연동)
- 튜토리얼 보상
- 휴면 매물 회수 (30일+ 데이터 쌓인 후)
- 푸시 알림

#### Phase 3: 출시 3개월 후

- 스킨 / 칭호 판매 시스템
- 랭킹 시스템 (자산/거래량/매물 시세)
- 운영 대시보드 (발행/회수 모니터링)
- 신고 시스템
- 친구 초대 보상 (안정화 후 검토)

### 11.2 출시 전 별도 진행 (코딩 외)

| 항목 | 내용 |
|---|---|
| 퍼블리시티권 정책 | 변호사 자문, 거부 토글, 신고 시스템 설계 |
| soop 약관/API 검토 | 합법적 데이터 수급 방법 결정 |
| 미성년 스트리머 보호 | 정책 수립 후 출시 |
| 베타 테스터 모집 | 클로즈 베타 → 오픈 베타 |

### 11.3 MVP 데이터 수급 전략

soop 공식 API 연동은 시간이 걸릴 수 있으므로, MVP 단계는 **유저 입력 기반**으로 시작.

- 유저가 직접 닉네임/ID를 입력 → 시스템이 매물로 등록
- 중복 검사: 이미 등록된 닉네임은 기존 매물로 안내
- 정식 출시 시점에 soop API 연동으로 전환 (검색 자동완성 등)

### 11.4 개발 단계별 순서

#### Step 1: 프로젝트 기반 구축

- GitHub 레포지토리 생성 (`streamauction`)
- 폴더 구조 잡기
- Firebase 프로젝트 생성 + 콘솔 설정
- Firebase Authentication 활성화 (Anonymous, Google)
- Firestore 활성화 + 보안 규칙 초안
- 기본 HTML/CSS 골격
- GitHub Pages 배포 테스트

#### Step 2: 인증 시스템

- 익명 로그인
- 구글 로그인
- 로그인 상태 UI 표시
- 익명 → 구글 전환 로직
- 가입 보너스 지급 (Cloud Function)
- 마이페이지 기본 (자산 표시)

#### Step 3: 매물 시스템

- 매물 등록 (유저 입력 기반)
- 매물 검색 페이지
- 매물 상세 페이지 (시세, 보유자, 거래 내역)
- 미보유 매물 구매 기능
- 보유 매물 표시 (마이페이지)
- 보유 한도 체크

#### Step 4: 거래 시스템

- 거래 제의 송신 (가격 입력 → 제의 생성)
- 거래 제의 수신 알림
- 거래 제의 승인 (자산 이동, 시세 갱신, 수수료 회수)
- 거래 제의 거절
- 거래 제의 만료 처리 (72시간)
- 즉시 손절 기능
- 거래 내역 기록

#### Step 5: 어뷰징 방어

- 동일 디바이스/IP 거래 차단 (Cloud Function 검증)
- 신규 24시간 손절 차단
- 매물 구매 후 24시간 손절 쿨다운
- Firestore 보안 규칙 강화

#### Step 6: 마무리

- 익명 유저 거래 차단 UI 처리
- 익명 → 구글 전환 유도 모달
- 에러 처리, 로딩 상태
- 모바일 반응형 점검
- 베타 테스트

### 11.5 Firebase 요금제 필요사항

| 요금제 | 사용 가능 | 비고 |
|---|---|---|
| Spark (무료) | 인증, Firestore, Hosting | Cloud Functions 사용 불가 |
| **Blaze (종량제)** | **모든 기능 + Cloud Functions** | **필수** (무료 한도 있어 초기 비용 거의 없음) |

**Cloud Functions 필수 이유:** 거래 처리, 가입 보너스 지급, 시세 계산 등 핵심 로직은 클라이언트에서 실행 시 어뷰징 가능. 반드시 서버에서 검증해야 함.
